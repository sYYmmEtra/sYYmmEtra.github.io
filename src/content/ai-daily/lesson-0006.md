# 📅 2026-07-10 · 轨道C · 混合专家 MoE（稀疏激活与路由）· 深度 L2

> **TL;DR**：MoE 把 Transformer 里那个最占参数的 FFN 子层，换成"一堆 FFN 专家 + 一个路由器"，每个 token 只走其中 top-K 个专家。于是**总参数量（决定容量/记忆）**和**每 token 激活参数量（决定算力/延迟）**被解耦——你能把模型做到 671B 却只花 37B 的计算。代价是训练时专家负载会"塌方"（routing collapse），整个 MoE 工程史，本质就是一部**怎么让 token 均匀分到专家身上、又不伤害模型质量**的斗争史。

---

## 一、核心概念精讲（L2）

### 0. 先把它放回昨天的地图上

我们在 07-06（轨道B）拆过 Transformer：每个 block = **注意力子层 + FFN 子层**（前馈网络，也叫 MLP）。注意力负责"token 之间互相看"，FFN 负责"对每个 token 单独做一次非线性变换、存放知识"。

一个关键事实：**FFN 占了 Transformer 参数量的约 2/3**。一个 hidden size `d=4096` 的标准 FFN 是 `d → 4d → d`（中间层放大 4 倍），光这一层就有约 `2 × 4 × d² ≈ 1.3 亿` 参数，而注意力的 QKVO 投影只有 `4 × d² ≈ 0.67 亿`。所以，**想在"不增加每步算力"的前提下塞进更多参数，最划算的下手点就是 FFN**。这正是 MoE 的切入点：不动注意力，只把 FFN 变"稀疏"。

### 1. L1 直觉：从"一个全科医生"到"一间专科医院"

**稠密模型（Dense）**：每个 token 来了，都要过同一个巨大的 FFN——好比不管你是感冒还是骨折，都去看同一个"全科医生"，他脑子里装着所有科室的知识，每次接诊都要调动全部。

**MoE**：把这个全科医生拆成 N 个"专科医生"（专家，expert，每个就是一个小 FFN），门口加一个**分诊台（路由器 / 门控网络，router / gating network）**。token 进来，分诊台看一眼，只把它转给最相关的 K 个专家（通常 K=1 或 2），其余专家这一步完全不工作。

一句话心智模型：**MoE = 条件计算（conditional computation）**。计算图不再对所有 token 一视同仁，而是"每个输入激活网络的不同子集"。

> ⚠️ **头号误区**：专家 ≠ 人类可理解的领域（"这个管数学，那个管代码"）。实测中专家的分工是**涌现的、碎片化的**，往往按 token 层面的统计规律划分（某些专家偏爱标点、某些偏爱特定 subword），几乎无法用"学科"去解释。别把 expert 想象成 MoE 版的"学科老师"。
>
> ⚠️ **二号误区**：MoE ≠ 集成学习（ensemble）。集成是"多个模型各自给完整答案，再投票平均"；MoE 是"一个模型内部，每一层都稀疏地挑几个子模块协同算一次前向"。方向完全不同。

### 2. L2 机制：一个 MoE 层到底怎么算

把原来的一个 FFN 子层，替换成如下结构。设输入是某个 token 的隐藏向量 `x ∈ ℝ^d`：

**Step 1 · 路由打分**。一个极小的线性层（门控权重 `W_g ∈ ℝ^{d×N}`，N=专家数）给出每个专家的亲和度：

```
s = softmax(x · W_g)      # s ∈ ℝ^N，s_i = token 对第 i 个专家的偏好分
```

**Step 2 · Top-K 选择**。只保留分数最高的 K 个专家，其余置零（这一步就是"稀疏"的来源）：

```
K_idx = TopK(s, k)                     # 选出的专家下标集合
g_i   = s_i / Σ_{j∈K_idx} s_j          # 对被选中的 K 个重新归一化，作为加权系数
                                        #（有些实现直接用 s_i，不重归一化）
```

**Step 3 · 只跑被选中的专家，再加权求和**：

```
y = Σ_{i ∈ K_idx}  g_i · Expert_i(x)   # Expert_i 就是第 i 个 FFN
```

关键点：**Σ 只在被选中的 K 个专家上进行**。假设 N=64、K=2，那么每个 token 实际只做了 `2/64 = 3%` 的 FFN 计算量，却能从 64 个专家的"知识库"里取用。这就是"总参数大、激活参数小"的机械原理。

用具体数字对照 DeepSeek-V3：**总参 671B，但每 token 只激活 37B**——因为绝大多数参数分散在几百个专家里，任一 token 只碰其中一小撮。

### 3. 为什么会"塌方"——MoE 训练的核心痛点

理想很美好，现实是：**路由器会作弊**。训练初期某几个专家碰巧被多选了几次 → 它们得到更多梯度、学得更好 → 于是被选得更多 → 形成正反馈。最后的结局叫 **路由塌方（routing collapse）**：绝大多数 token 挤向少数几个专家，剩下的专家几乎从不被激活、等于白占参数。这不仅浪费容量，还让"稀疏"名存实亡。

更麻烦的是**硬件层面**：真实训练里专家分布在不同 GPU 上（专家并行，expert parallelism）。每个 GPU 有一个固定的**容量（capacity）**——最多接收多少个 token。为防止负载不均导致某张卡爆内存，工程上给每个专家设**容量上限 `capacity = capacity_factor × (tokens / N)`**。超出上限的 token 会被**直接丢弃（token dropping）**——它这一层的 FFN 输出变成 0（只剩残差连接把它带过去）。负载越不均，被丢的 token 越多，质量越差。

所以"负载均衡"不是锦上添花，是**MoE 能不能训得动**的生死线。

### 4. 三代负载均衡方案（今天的重点）

**方案一 · 辅助损失（auxiliary loss，2017 起的标准做法）**
在主损失外，额外加一项惩罚"负载不均"的损失。经典形式（Switch Transformer）：

```
L_aux = α · N · Σ_i (f_i · P_i)
```

其中 `f_i` = 实际被路由到专家 i 的 token 占比（这一项不可导，来自计数），`P_i` = 路由器给专家 i 的平均 softmax 概率（可导）。两者相乘再求和：当某专家又被多选（`f_i` 高）、路由器还继续给它高概率（`P_i` 高）时，这一项就大，梯度会把 `P_i` 压下去。`α` 是系数。

> **它为什么有效又有害**：有效，因为它直接给"偏心的路由器"施加反向梯度。有害，因为这是一股**干扰梯度（interference gradient）**——它和"把语言建模做好"这个主目标无关，纯粹为了均衡而扭曲参数。`α` 调大则均衡但掉智力，调小则智力保住但会塌方。这个"跷跷板"极难调，是 MoE 长期的痛。

**方案二 · 无辅助损失的偏置法（Loss-Free Balancing，DeepSeek 2024，现已成主流标准）**
思路极其巧妙：**不用梯度去纠偏，改用一个人工维护的偏置项**。给每个专家一个偏置 `b_i`，**只在 Top-K 选择时加上、不参与最终加权、也不回传梯度**：

```
选择依据：TopK(s_i + b_i, k)     # 用 s+b 来"选"
加权仍用：g_i ∝ s_i              # 用原始 s 来"算"，保证不污染前向
```

然后每个训练步后，看哪些专家过载、哪些空闲，**手动**调 `b`：过载的专家 `b_i` 调低（下一步少选它），空闲的调高（多选它）——像空调恒温器一样动态维持均衡。

> **为什么这是一大进步**：它把"均衡"这件事从**梯度域**移到了**离散的偏置调整**上，因此**完全不产生干扰梯度**，模型质量的上限被抬高了。这也是 DeepSeek-V3 采用的方案。2025-12 还有工作（见下方论文）把它证明成一个**原始-对偶（primal-dual）**优化过程，给出了对数级 regret bound——说明这个"土办法"背后有扎实的理论。

**方案三 · 结构性方案（DeepSeekMoE 的两个招式，可与上面组合）**
- **细粒度专家（fine-grained experts）**：不搞 8 个大专家，而是把每个专家再切小、搞 64 个小专家、每次选更多个（如选 8）。组合数从 C(8,2) 暴涨到 C(64,8)，专家分工能更精细、更容易特化。
- **共享专家（shared experts）**：留 1~2 个专家**对所有 token 恒定激活**，专门吸收"通用知识"（语法、常识），让其余路由专家腾出手去学"专门知识"，减少冗余。

### 5. 🔬 完整走一遍：一个 token 穿过 MoE 层

设 N=4 个专家，K=2，某 token 的隐藏向量 `x` 进来。

1. **路由打分**：`x·W_g` 得到 logits `[2.0, 0.5, 1.5, -1.0]`，过 softmax →
   `s = [0.52, 0.12, 0.31, 0.03]`（专家 0 最受青睐，专家 3 几乎被无视）。
2. **Top-2 选择**：选中专家 **0 和 2**（分数 0.52、0.31）。专家 1、3 这一步**完全不计算**。
3. **重归一化权重**：`g_0 = 0.52/(0.52+0.31) = 0.63`，`g_2 = 0.31/0.83 = 0.37`。
4. **跑被选专家**：`Expert_0(x) = u`，`Expert_2(x) = v`（各是一次 `d→4d→d` 的 FFN 前向）。
5. **加权求和**：`y = 0.63·u + 0.37·v`。
6. **接回主干**：`y` 经残差 + LayerNorm，送入下一层。

**现在看负载均衡怎么介入**：假设这一 batch 里专家 0 已经接了太多 token、到了容量上限，而我们用的是 **Loss-Free 偏置法**——那么专家 0 的偏置 `b_0` 早已被调低（比如 −0.4）。于是第 2 步的选择依据变成 `s+b = [0.12, 0.12, 0.31, 0.03]`，这个 token 就会**改选专家 2 和 1**，把负载让出去；但第 3 步算加权时**仍用原始 `s`**，前向计算不受偏置污染。这就是"用偏置调度、用原分打分"的精髓。

> ⚠️ **三号误区**：MoE 省的是**算力（FLOPs / token）**，不省**显存**。671B 参数即便只激活 37B，那 671B 也**全部要装进显存**（分片在多卡上）。而且专家分散在多卡，每层都要做一次昂贵的 **all-to-all 通信**把 token 发到对应专家所在的卡、再收回来。所以 MoE 未必"wall-clock 更快"——它是**在同等算力预算下换更高的模型质量**，通信和显存是它真实的账单。

## 三、🔁 旧知回顾（间隔重复日 · 第 6 天）

- **Q1（07-06 注意力）**：自注意力公式里为什么要除以 `√d`？不除会怎样？
- **Q2（07-07 测试时计算）**：并行式 test-time compute（采 N 条候选 + 验证器选优）里，为什么"坏验证器"会导致增大 N 反而收益下降甚至反转？
- **Q3（07-09 优化器）**：Adam 里的"二阶矩" `v` 起什么作用？为什么它能实现"每个参数不同的有效学习率"？

<details><summary>点开看答案</summary>

- **A1**：`QKᵀ` 是 d 维向量点积，其方差随维度 d 线性增长，d 大时点积数值会很大，softmax 落入饱和区（梯度趋近 0、输出接近 one-hot）。除以 `√d` 把点积方差归一化回 ~1 的量级，让 softmax 保持在梯度健康的区间。不除的话，长向量下注意力会过早"硬化"、训练难以进行。

- **A2**：并行式方法靠"多采样扩大覆盖率 coverage、再由验证器挑对的那条"。coverage 随 N 增长（对数近似），但真正被选中的取决于验证器质量。若验证器有系统性偏差，它会**稳定地偏爱某些错误答案**；N 越大，正好落在这些"高分错误"上的样本越多，被选中的概率反而升高——于是曲线见顶后掉头向下。收益上限由验证器而非采样数决定。

- **A3**：`v` 是**梯度平方的滑动平均**，衡量每个参数近期梯度的"大小/波动"。更新时用 `m̂ / (√v̂ + ε)`：梯度一贯很大的参数被大 `√v` 除小、步子收敛；梯度小的参数被小 `√v` 放大、步子加大。等于把每个参数的更新量都归一化到 ±1 量级，实现**逐参数自适应的有效学习率**，这也是 Adam 对学习率不那么敏感的原因。

</details>
